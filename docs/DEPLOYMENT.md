# Deployment

The stack splits across three hosts, each chosen for what it does well:

| Piece | Host | Why |
| --- | --- | --- |
| Frontend | Vercel | Static bundle on a CDN; previews per pull request |
| API | Render (Docker) | Socket.io needs a long-lived process, not serverless functions |
| Database | MongoDB Atlas | Managed, free tier is sufficient, independent of the app host |

Everything is reproducible from committed config: [`render.yaml`](../render.yaml)
and [`frontend/vercel.json`](../frontend/vercel.json).

---

## 1. MongoDB Atlas

1. Create a free **M0** cluster.
2. **Database Access** → add a user with *Read and write to any database*.
3. **Network Access** → Render does not publish fixed egress IPs on the free
   plan, so allow `0.0.0.0/0`. The database is still protected by SRV
   credentials and TLS; on a paid plan, use a fixed egress IP instead.
4. Copy the SRV connection string and append the database name:

   ```
   mongodb+srv://<user>:<password>@<cluster>.mongodb.net/bharadwaj-cafe?retryWrites=true&w=majority
   ```

## 2. API on Render

1. **New → Blueprint**, point it at this repository. Render reads `render.yaml`.
2. Fill in the variables marked `sync: false`:

   | Variable | Value |
   | --- | --- |
   | `MONGO_URI` | the Atlas string from step 1 |
   | `CLIENT_URL` | your Vercel URL, e.g. `https://bharadwaj-cafe.vercel.app` |
   | `EMAIL_HOST` / `EMAIL_PORT` / `EMAIL_USER` / `EMAIL_PASS` | SMTP credentials |

   `JWT_SECRET` is generated by Render on first deploy. **Do not regenerate
   it** — every issued token becomes invalid the moment it changes.

3. Deploy. Render builds `backend/Dockerfile` and waits for `/api/health`.

`CLIENT_URL` is the CORS allowlist and accepts a comma-separated list, so
preview and production origins can coexist:

```
https://bharadwaj-cafe.vercel.app,https://bharadwaj-cafe-git-main-you.vercel.app
```

## 3. Frontend on Vercel

1. **Add New → Project**, import the repository, set **Root Directory** to
   `frontend`. `vercel.json` supplies the rest.
2. Add one environment variable:

   ```
   VITE_API_URL = https://bharadwaj-cafe-api.onrender.com/api
   ```

   Vite inlines `VITE_*` at build time, so **changing it requires a redeploy**,
   not just a restart.
3. Deploy, then paste the resulting URL back into Render's `CLIENT_URL`.

## 4. Seed the database

From a machine with the Atlas URI available:

```bash
cd backend
MONGO_URI="<atlas-uri>" npm run seed

MONGO_URI="<atlas-uri>" \
ADMIN_EMAIL="you@example.com" \
ADMIN_PASSWORD="<at least 12 characters>" \
npm run create-admin
```

## Order of operations

There is a deliberate circular dependency: the API needs `CLIENT_URL` and the
frontend needs `VITE_API_URL`. Resolve it in this order:

1. Deploy the API with a placeholder `CLIENT_URL`.
2. Deploy the frontend using the real API URL.
3. Update `CLIENT_URL` on Render with the real frontend URL. Render restarts
   automatically.

## Known constraints of the free tier

- **Render sleeps after ~15 minutes idle.** The first request afterwards takes
  30–60 seconds. Worth stating on the demo page so a recruiter clicking a cold
  link does not assume the app is broken.
- **Sleeping also drops WebSocket connections.** Socket.io reconnects on its
  own, but live order updates pause while the service is asleep.
- **Atlas M0 has no backups.** Fine for a demo; not for real customer data.

## Verifying a deployment

```bash
curl -s https://<api-host>/api/health          # {"status":"OK",...}
curl -s https://<api-host>/api/menu | head -c 200

# CORS actually configured for the deployed frontend:
curl -si -X OPTIONS https://<api-host>/api/menu \
  -H "Origin: https://<frontend-host>" \
  -H "Access-Control-Request-Method: GET" | grep -i access-control-allow-origin
```

## Rollback

- **Vercel** — Deployments tab → *Promote to Production* on the last good build.
- **Render** — Events tab → *Rollback* on the previous deploy.

Neither reverts a database migration, so schema changes need to stay
backward-compatible for at least one release.
